# План доработок SnapShot

## Цель

Довести галерею до стабильного production-ready SPA: улучшить UX, обработку
ошибок, доступность, расширяемость и качество проверок.

Отдельная цель: развивать проект так, чтобы FSD был оправдан реальными
слайсами, моделями и межслайсовыми зависимостями, а не только структурой папок.

## Правила работ

- Сохранять текущую FSD-структуру: `app -> pages -> widgets -> features -> entities -> shared`.
- Не добавлять barrel `index.ts` без отдельной причины.
- Для пользовательских сценариев добавлять тесты рядом с изменяемым модулем.
- После изменений запускать `yarn typecheck`, `yarn lint`, `yarn test`; для крупных изменений дополнительно `yarn build`.

## Последовательный план

### Шаг 1. Стабилизировать базовый UX

1. Разделить состояния галереи на понятные компоненты: `Loading`, `Empty`, `Error`, `ImageFallback`.
2. Улучшить empty/error states: пустая выдача, ошибка сети, отсутствующий Flickr API key, ошибки Flickr API.
3. Добавить fallback для битых изображений: скрывать skeleton и показывать нейтральный placeholder.
4. Проверить контрастность текущей палитры, `disabled`-состояния, keyboard/focus states поиска, навигации и карточек фото.
5. Обернуть основной контент в семантический `main`, а корневой `Header`-контейнер заменить на `header`.

### Шаг 2. Укрепить API-слой и тесты

1. Добавить нормализацию cache key для поиска: `trim` + единый регистр, чтобы `bird` и `Bird` не создавали разные записи.
2. Расширить типы Flickr-ответа: хранить `page`, `pages`, `perpage`, `total`.
3. Добавить конфиг размера изображений Flickr для переключения `thumbnail`/`medium` без ручной правки URL.
4. Добавить тесты на `ApiError`, невалидный JSON и Flickr `stat: fail`.
5. Добавить тесты на маршруты: редирект `/`, категории, `/search/:searchInput`, 404.

### Шаг 3. Доработать поиск и ссылки

1. Сохранять последний поисковый запрос в input при открытии страницы поиска.
2. Добавить share-ссылку на текущий поиск или категорию.
3. Добавить фильтры и сортировку: ориентация, релевантность/дата, лицензия, цвет, safe search.
4. Хранить состояние фильтров в URL, чтобы поиск можно было открыть по ссылке.

### Шаг 4. Добавить дозагрузку выдачи

1. Реализовать кнопку `Load more` или пагинацию на основе `page/pages/total`.
2. Добавить дедупликацию фото между страницами.
3. Сохранять позицию пользователя при возврате к выдаче.
4. Рассматривать virtual scroll только после появления реальных проблем с производительностью.

### Шаг 5. Сделать карточки фото интерактивными

1. Сделать карточку фото кликабельной.
2. Добавить лайтбокс или страницу `/photo/:id`.
3. Подтянуть детали через `flickr.photos.getInfo`: EXIF, автор, размеры, sharing.
4. Добавить навигацию вперед/назад между фото внутри текущей выдачи.

### Шаг 6. Подготовить FSD к росту домена

1. Выделить единый доменный тип фото в `entities/photo`.
2. Добавить адаптеры провайдеров для Flickr, Unsplash и Pexels.
3. Держать provider-specific код ниже доменной модели, чтобы UI работал с единым типом.
4. После этого расширять фильтры и детали фото через доменные контракты, а не через прямые URL/API-форматы.

### Шаг 7. Добавить пользовательские сценарии

1. Добавить локальное избранное в `localStorage`: `entities/favorite`, `features/like-photo`.
2. Если нужно синхронизировать пользовательские данные, добавить Flickr OAuth: `entities/user`, `features/auth`, хранение токенов, refresh, protected routes, guards.
3. После auth добавить синхронизацию избранного: оптимистичные апдейты и инвалидация кэша.
4. Добавить альбомы пользователя CRUD: `entities/album`, `features/manage-album`, `widgets/album-board`, `pages/album`.

### Шаг 8. Расширить платформенные возможности

1. Добавить дизайн-систему в `shared/ui`: Button, Input, Modal, Skeleton.
2. Добавить Storybook и визуальные тесты, когда компонентов станет достаточно для повторного использования.
3. Добавить i18n, темизацию и design tokens: `app/providers`, `shared/config`, `shared/ui`.
4. Добавить PWA/offline-режим: service worker, кэш изображений, очередь действий, IndexedDB, офлайн-синхронизация.

### Шаг 9. Закрепить качество в CI

1. Добавить smoke-тест production build через `yarn build` в CI, если его еще нет в workflow.
2. Проверить, что CI запускает `yarn typecheck`, `yarn lint`, `yarn test`.
3. Зафиксировать обязательную проверку mobile и desktop ширины для UI-изменений.

## Definition of Done

- Изменение покрыто тестами или явно указана причина, почему тест не нужен.
- Нет ошибок `yarn typecheck`, `yarn lint`, `yarn test`.
- Для крупных изменений проходит `yarn build`.
- UI проверен на mobile и desktop ширине.
- Ошибки API не ломают страницу и показывают понятное состояние пользователю.
